## What is RPG Chronicle

A system that records tabletop RPG sessions (via Discord/Craig Bot), transcribes them, extracts entities and relationships (characters, locations, factions), indexes everything into a knowledge graph, and allows natural language queries about what happened in the campaign.

**Data science analogy**: think of an ETL pipeline where the source is audio, the transformation is NLP/LLM, and the destination is a vector store + graph. The final "data product" is a chatbot that answers questions about the campaign.

## Core Concepts (for someone coming from Data Science)

### What is a "monorepo" and why use one

**Concept**: instead of separate repositories for each part of the system (frontend, backend, bot), everything lives in a single repository with separate folders.

**Analogy**: like having an ML project where `notebooks/`, `src/`, `models/`, `api/` are all in the same repo instead of separate repos. Keeps everything synchronized.

**When to decide this**: when you have multiple services that share types, configs, or need to be deployed together. For a personal project, monorepo is almost always the right answer.

### What is a "package manager" and why are there many

**Concept**: a tool that installs and manages your code's dependencies. Like `pip` or `conda` for Python, JavaScript has `npm`, `yarn`, and `pnpm`.

**Why pnpm**: faster and uses less disk than npm/yarn (uses symlinks instead of copying packages). Current standard for monorepos.

**Why UV for Python**: same idea — it's the "pnpm of Python". Resolves dependencies 10-100x faster than pip/poetry. Uses the same `pyproject.toml` you already know.

### What is "authentication" vs "authorization"

**Authentication** = "who are you?" (login with Discord)
**Authorization** = "what can you see/do?" (only campaign members see its data)

**Why decide this**: every multi-user app needs to solve both problems. The decision is: implement from scratch (complex, security-bug-prone) or use a ready-made service (Supabase Auth).

### What is a "queue" (job queue)

**Concept**: when a task takes too long (transcribing 3h of audio), you don't want the user waiting. The queue receives the request, responds "ok, I'll process it", and a separate worker executes in the background.

**Analogy**: like submitting a job in Spark/Airflow — you don't wait for it to finish, you get a job_id and check status later.

**Why decide this**: without a queue, the web server is blocked processing audio and can't respond to other requests. With a queue, the server stays fast and heavy processing happens in parallel.

### What is "Row Level Security" (RLS)

**Concept**: rules in the database that automatically filter data based on who's asking. Even if your code has a bug that forgets to filter by campaign, the database won't return data the user shouldn't see.

**Analogy**: imagine if every SQL query you write had an automatic, invisible `WHERE user_id = current_user`. You can't forget to add it because the database adds it for you.

**Why decide this**: it's a security layer that protects against your own bugs. Not necessary in the MVP (only you use it), but impressive in a portfolio.

### What is "Server Component" vs "Client Component"

**Concept in Next.js**: code that runs on the server (fetches data, no JavaScript in the browser) vs code that runs in the user's browser (buttons, forms, interactive state).

**Analogy**: Server Component is like a static report generated by a script — the final result is pure HTML. Client Component is like an interactive Streamlit dashboard — needs code running in the browser.

**Simple rule**: if the component only displays data, it's Server. If the user interacts (clicks, types, drags), it's Client.

### What is response "streaming"

**Concept**: instead of waiting for the LLM to generate the entire response and then showing it, each token is sent to the frontend as it's generated. The user sees the response "being written" in real time.

**Why it matters**: LLM responses can take 5-15 seconds. Without streaming, the user stares at a blank screen. With streaming, they see immediate progress.

## Database Schema — By Phase

### Why versioned migrations?

**Concept**: instead of editing the database manually, you write numbered SQL scripts that are applied in order. This guarantees anyone (or CI) can recreate the database exactly as it is.

**Analogy**: like versioning your notebooks — instead of "run this cell, then that one", you have a reproducible script.

### Migration 001 — MVP (Phase 0)

```sql
create extension if not exists vector;

create table campaigns (
  id                  uuid primary key default gen_random_uuid(),
  discord_server_id   text not null unique,
  name                text not null,
  created_at          timestamptz default now(),
  settings            jsonb default '{}'
);

create table campaign_members (
  campaign_id         uuid references campaigns(id) on delete cascade,
  discord_user_id     text not null,
  role                text check (role in ('gm', 'player')) default 'player',
  joined_at           timestamptz default now(),
  primary key (campaign_id, discord_user_id)
);

create table sessions (
  id                  uuid primary key default gen_random_uuid(),
  campaign_id         uuid references campaigns(id) on delete cascade,
  session_number      integer not null,
  title               text,
  recorded_at         timestamptz,
  duration_seconds    integer,
  status              text check (status in (
                        'recording', 'processing', 'indexing', 'ready', 'error'
                      )) default 'recording',
  audio_path          text,
  transcript_path     text,
  summary             jsonb,
  error_message       text,
  created_at          timestamptz default now(),
  unique(campaign_id, session_number)
);

create table session_chunks (
  id                uuid primary key default gen_random_uuid(),
  session_id        uuid references sessions(id) on delete cascade,
  campaign_id       uuid references campaigns(id) on delete cascade,
  content           text not null,
  chunk_index       integer not null,
  speaker           text,
  timestamp_start   integer,
  timestamp_end     integer,
  embedding         vector(1536),
  created_at        timestamptz default now()
);

create index on session_chunks (session_id, chunk_index);
```

### Migration 002 — Entities and Relations (Phase 1)

```sql
create table entities (
  id                      uuid primary key default gen_random_uuid(),
  campaign_id             uuid references campaigns(id) on delete cascade,
  name                    text not null,
  type                    text check (type in (
                            'character', 'location', 'faction',
                            'item', 'event', 'secret', 'concept'
                          )),
  description             text,
  first_seen_session_id   uuid references sessions(id),
  last_seen_session_id    uuid references sessions(id),
  metadata                jsonb default '{}',
  embedding               vector(1536),
  created_at              timestamptz default now(),
  unique(campaign_id, name, type)
);

create table entity_relations (
  id              uuid primary key default gen_random_uuid(),
  campaign_id     uuid references campaigns(id) on delete cascade,
  source_id       uuid references entities(id) on delete cascade,
  target_id       uuid references entities(id) on delete cascade,
  relation_type   text not null,
  description     text,
  confidence      float default 1.0,
  session_id      uuid references sessions(id),
  created_at      timestamptz default now()
);

create table chat_queries (
  id                uuid primary key default gen_random_uuid(),
  campaign_id       uuid references campaigns(id) on delete cascade,
  user_discord_id   text not null,
  query             text not null,
  response          text,
  sources           jsonb,
  latency_ms        integer,
  feedback          smallint,
  created_at        timestamptz default now()
);

create index on entities (campaign_id, type);
create index on entity_relations (campaign_id, source_id, target_id);
```

### Migration 003 — RLS (Phase 2)

```sql
alter table campaigns enable row level security;
alter table sessions enable row level security;
alter table entities enable row level security;
alter table entity_relations enable row level security;
alter table session_chunks enable row level security;
alter table chat_queries enable row level security;

create or replace function is_campaign_member(p_campaign_id uuid)
returns boolean as $$
  select exists (
    select 1 from campaign_members
    where campaign_id = p_campaign_id
    and discord_user_id = (auth.jwt() ->> 'sub')
  );
$$ language sql security definer;

create policy "members_read_campaigns" on campaigns
  for select using (is_campaign_member(id));

create policy "members_read_sessions" on sessions
  for select using (is_campaign_member(campaign_id));

create policy "members_read_entities" on entities
  for select using (is_campaign_member(campaign_id));

create policy "members_read_chunks" on session_chunks
  for select using (is_campaign_member(campaign_id));
```

### Migration 004 — Performance Indexes (Phase 2+)

```sql
-- ivfflat needs existing data to work well (minimum ~1000 rows)
-- Only add when you have enough volume
create index on entities using ivfflat (embedding vector_cosine_ops) with (lists = 100);
create index on session_chunks using ivfflat (embedding vector_cosine_ops) with (lists = 100);
```

## Technical Decisions — Quick Reference

### Auth: Supabase Auth (not NextAuth)
- Native Discord OAuth provider in Supabase — dashboard config, zero code
- Session managed by Supabase SDK
- `anon key` on frontend (subject to RLS), `service_role key` only in pipeline

### Why is the pipeline separate from Next.js?
- **Analogy**: like separating your exploration notebook from your production script. The frontend serves pages fast; the pipeline processes audio for 30 minutes. If they were together, one would block the other.
- Independent scaling (can run on a machine with more RAM/GPU)
- Different languages (Python for ML/NLP is natural; Next.js is JavaScript)

### ARQ vs Celery (Phase 1+)
- ARQ is asyncio-native (our entire pipeline is async — like using `asyncio` instead of threads)
- Redis is already needed for cache
- Celery is the "Airflow" of queues — powerful but way too complex for this case

### Repository Pattern
- **Concept**: instead of scattering SQL queries throughout the code, centralize them in one place. Want to change how you fetch sessions? Change one file.
- **Analogy**: like having a `data_loader.py` that encapsulates all data access — the rest of the code doesn't know if it comes from CSV, database, or API.

### UV vs Poetry
- 10-100x faster dependency resolution
- Same standard `pyproject.toml`
- `uv sync`, `uv add`, `uv run` — intuitive commands (equivalent to `pip install -r`, `pip install <pkg>`, `python <script>`)

## What NOT to Do (anti-patterns)

- **Don't implement auth from scratch** — use Supabase Auth. Custom auth is the #1 source of vulnerabilities.
- **Don't create a design system** — use shadcn/ui or whatever Stitch generates. Focus on the product, not buttons.
- **Don't optimize before measuring** — ivfflat indexes, caching, etc. only when you have data and latency metrics.
- **Don't use Neo4j** — pgvector + LightRAG is enough. Neo4j adds infra, cost, and complexity without clear benefit at this scope.
- **Don't deploy before Phase 5** — run locally until you have something presentable. Premature deploy is a distraction.
- **Don't try to process real audio in Phase 0** — use manually pasted text or a .txt file as input.
- **Don't over-engineer** — if a function is 10 lines and readable, it doesn't need a class, a factory, or an abstract interface.
